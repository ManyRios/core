# 系统模型

## 目的

本节说明请求如何穿过 Veil，以及每个运行角色可以看到什么、控制什么。

## 适用场景

- 你需要端到端请求链路
- 你在核对信任或可见性边界
- 你想理解不同角色分别拥有哪部分运行职责

## 角色

当前已部署的运行时角色是 Consumer、Relay、Provider 和 Bootstrap。Claw 是市场主路径上的目标运维自动化角色，不是可选的边缘工具。

### Consumer

- 提供本地 HTTP 网关
- 对请求进行签名和加密
- 执行本地 API 认证、预算和超时控制

### Relay

- 接收 Consumer 与 Provider 连接
- 校验请求签名
- 应用限流
- 把请求路由到 Provider
- 记录见证记录

### Provider

- 解密密封后的请求载荷
- 调用上游 AI API
- 返回单次或流式响应
- 管理本地并发与凭证

### Bootstrap

- 存储 Relay 注册与心跳
- 向客户端返回打分后的 Relay 列表

### Claw

- 接收运维者的接入网络、卖出容量等意图
- 自动化节点接入和运行时启动
- 应用定价与风险策略
- 编排健康检查、暂停、恢复和故障处理流程

## 接入模型

- Consumer 通过本地网关访问 AI，而不是让每个工具都直接集成每个上游账户
- Provider 向网络路径提供可路由的执行能力
- Relay 在不进入解密边界的前提下撮合供给并记录见证记录
- Claw 通过低人工干预自动化减少节点运维负担，让 Provider 容量可以被持续发布和维护

## 请求链路

1. Consumer 接收 OpenAI 兼容请求。
2. Consumer 选择 Provider 并加密`内层载荷（inner payload）`。
3. Relay 验签并转发请求。
4. Provider 解密并调用上游模型。
5. Relay 转发结果并记录见证记录。
6. Consumer 将结果转换成客户端可消费的输出。

## 运维路径

1. 运维者表达接入网络或出售容量的意图。
2. Claw 校验本地前置条件、凭证和策略护栏。
3. Claw 发现或选择可用的 Relay 基础设施。
4. Claw 启动或协调 Provider 运行时状态。
5. Claw 让定价、容量发布和健康行为持续符合运维策略。
6. 见证记录和结算系统消费由此产生的用量证据、定价版本和报价元数据。

## 可见性边界

| 角色 | 可见内容 |
|------|---------|
| Consumer | 本地提示词、本地响应、本地钱包、本地预算 |
| Relay | 请求元数据、路由元数据、见证记录元数据 |
| Provider | 提示词明文、模型参数、上游用量 |
| Bootstrap | 仅 Relay 元数据 |
| Claw | 运维者意图、本地运行时状态、策略状态和健康结果 |

Relay 不应访问提示词明文。Provider 不应获得超出执行所需范围的 Consumer 原始身份上下文。

## 隐私定位

Veil 旨在提供无账户接入和隐私保护式路由，也就是把可见性拆分到不同角色上。它并不意味着 Veil 能对流量分析、端点失陷、上游可见性或节点串谋提供完美匿名保证。

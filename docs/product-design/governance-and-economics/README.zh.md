# 治理与经济边界

## 目的

本节说明 Veil 如何同时处理两类经济设计：

- Consumer、Provider、Relay 之间的推理市场闭环
- maintainer、contributor 和 RBOB 之间的开源构建循环

## 分离原则

Veil 将推理流量和贡献账务放在不同域中处理。

## 当前运行时边界

- 当前运行时已经提供基于报价单位的预算保护、用量归一化、带签名的见证记录、可导出的审计数据，以及贡献账务能力
- 当前运行时还没有暴露终端支付轨道、Provider 收益分发流程，或 Relay 收益分发 API

## 报价语言与结算资产

- 运行时可以先使用报价单位做预算保护和确定性价格比较；当前报价语言是基于 USD 的估值
- 报价单位不是最终结算资产
- 即使先落地报价驱动的预算能力，crypto 原生支付与结算仍然保留在产品主线上

## 推理市场方向

- Veil 正在朝着这样的市场闭环构建：Consumer 为可用的 AI 接入付费
- Provider 因实际执行能力获得收益
- Relay 因路由、准入和见证记录服务获得收益
- 见证记录存在的意义之一，就是让用量和后续结算可以被对账

## 结算方向

- 结算应该来源于请求与见证记录证据，而不是不透明的旁路数据
- 定价与结算可以独立于底层传输演进
- 在接口和导出层面，报价金额与结算资产必须显式区分
- 后续支付轨道应当消费同一套见证记录与账务边界，而不是绕过它们

## 治理侧

- maintainer 负责评审和合并代码
- RBOB 记录贡献积分
- 保护模块需要更严格的评审

## 社区原则

Veil 是公开的开源项目。贡献者、评审者和 maintainer 都应该是项目表面的一部分，贡献记录也应当保持可归属、可审计。

## RBOB 的角色

RBOB 是贡献账本，不属于推理热路径。它的职责是记录带签名的贡献条目、余额、排行榜和可导出的 grant 记录。

## 账务规则

- 生产见证记录数据必须与贡献积分分离
- 推理收入与贡献奖励必须在数据模型层面保持可分离
- 贡献 grant 必须带签名
- 治理数据必须可核验、可导出

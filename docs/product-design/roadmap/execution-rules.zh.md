# 执行规则

## 目的

本页定义 Veil 在实施路线图时应遵循的顺序规则，避免路线图变成并行堆叠的愿望清单。

## 规划单元

每个路线图事项都应归入以下单元之一：

- `Foundation`：后续自动化依赖的基础运行时能力
- `Control`：协调市场的发现、策略和健康系统
- `Market`：容量发布、定价和卖方自动化
- `Settlement`：基于见证记录的账务和可支付输出
- `Privacy`：基础隐私保护式路由属于必选能力；更强隐私配置可分阶段推进，但不能阻塞标准市场闭环

## 硬依赖顺序

1. 先有运行时基线，再做 autopilot
2. 先有发现和健康系统，再做自动卖量
3. 先有定价和风险策略，再做自治市场发布
4. 先有见证记录完整性和定价快照，再做收益分发逻辑
5. 先跑通标准市场运营，再考虑强匿名配置

## 阶段门槛规则

### 运行时基线门槛

后续阶段依赖它之前，必须先具备：

- 稳定的 Consumer -> Relay -> Provider 请求链路
- 可重现的见证记录生成
- 有界的重试、超时和限流行为
- 运维者可见的健康状态和日志

### 发现门槛

低人工干预自动化启动前，必须先具备：

- Bootstrap 能返回可用的 Relay 集合
- Provider 能在多个 Relay 候选之间重连
- Relay 丢失后，本地运行时能在不重建环境的情况下恢复

### Autopilot 门槛

卖方自动化对外暴露前，必须先具备：

- 可持久化的运维者意图
- 接入流程能在暴露容量前校验前置条件
- 健康退化能自动触发暂停或恢复
- 自动化层不能绕过现有安全检查

### 市场控制门槛

只有满足这些条件，自治卖量才算完成：

- 定价策略是确定性的
- 风险包络能阻止不安全的卖量
- 报价发布和撤回可追踪
- 调价不会破坏见证记录账务

### 结算门槛

收益分发接口暴露前，必须先具备：

- 结算条目能由见证记录加定价快照重现
- 有重复结算保护
- 收益分发输出与贡献账务保持分离
- 导出流程可审计

## 路线图能力的完成定义

每个路线图能力上线时都必须同时具备：

- 明确的模块归属
- 接口和状态定义
- 失败处理规则
- 安全约束
- 测试用例
- 运维可见的状态或日志
- 对应文档更新

## 红线规则

- 在没有策略和健康门槛前，不得上线卖方自动化
- 在没有确定性结算输入前，不得上线收益分发工作流
- 不得让隐私承诺跑在系统实际保证前面
- 不得把 RBOB 和市场结算并入同一本账

## 推荐拆解顺序

1. 对齐文档和模块边界
2. 补齐缺失的基础代码
3. 增加由 Claw 托管的接入流程
4. 增加定价与风险策略
5. 增加结算和收益分发账务
6. 增加接近自治的市场运营
7. 之后再评估更强的隐私配置
